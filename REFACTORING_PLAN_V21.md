# ChainFlow_V21_Suite リファクタリング計画 (保留中)

## 目的
ChainFlow_V21_Suite全体における技術的負債を解消し、将来的な機能拡張やメンテナンスを容易にするためのアーキテクチャ改善計画。現在はリスク回避のため実行を保留しているが、着手する際はこの計画に沿って段階的に進める。

## フェーズ1: 共通ユーティリティのライブラリ化（ChainFlowUI）
各アプリケーションに散在している定型コード（ボイラープレート）を一つの共通パッケージに集約する。

- **対象コード**:
  - Windowsダークモードタイトルバー適用処理 (`ctypes.windll.dwmapi` による設定)
  - タスクバーAppID設定 (`ctypes.windll.shell32` による設定)
  - アプリ全体に適用しているダークテーマ用QSSスタイル（QScrollBar, QMessageBoxなどのデザイン定義）
- **メリット**:
  デザインの変更が1箇所で済み、新規アプリ追加時も数行のコードで統一されたUIが構築可能になる。

## フェーズ2: 巨大UIクラス（God Object）の責任分割
GUI構築ロジック、データモデル、ビジネスロジックが密結合しているメガクラスを分割する。

- **対象ファイル**:
  - `ChainFlowFiler_v21/widgets/main_window.py` (`ChainFlowFiler` クラス)
  - `ChainFlowDesigner/dtp_editor.py` (`DTPEditor` クラス)
  - `ChainFlowSearch/app_window.py` (`SearchWindow`, `SearchWidget` クラス)
- **アプローチ**:
  - **View層**: `setup_ui()` などのウィジェット配置ロジックのみを担当するクラスへ分離。
  - **Controller/Action層**: SlashMenuの処理や保存、エクスポートなどの各機能を個別のコマンドクラス、あるいは別モジュールへ移譲。
  - **メリット**: コードの可読性が格段に上がり、機能の追加やデバッグがしやすくなる。副次的なバグの発生率を低下させる。

## フェーズ3: Model / View アーキテクチャの導入
データリストの表示パフォーマンスと整合性を高める。

- **対象アプリ**:
  - `ChainFlowToDo`
  - `ChainFlowSearch`
- **アプローチ**:
  - `QVBoxLayout` に対して都度 `addWidget` や `deleteLater` を繰り返す手動DOM操作的な手法から脱却。
  - Qt標準の `QAbstractListModel` (または `QAbstractTableModel`) を継承した独自のモデルを作成し、内部データを一元管理。
  - `QListView` や `QTableView` にモデルをバインドして描画させる。
- **メリット**:
  数千件のタスクや検索結果が表示される場合でも、UIスレッドがブロックされずスクロールが滑らかになる。

## フェーズ4: バックグラウンド処理・エラー通知機能の統合
独立して動く各ツールの非同期処理をより安全にするための共通基盤。

- **アプローチ**:
  - `QThread` (Worker) を用いた重い処理（ファイル一括処理、検索、画像変換）において、途中でウィンドウが閉じられた場合の安全なキャンセル処理を全アプリで標準化。
  - 共通の「タスクプログレス＆ログマネージャー」UIコンポーネントを作成し、エラー時のスタックトレースや完了通知を洗練された形でユーザーに提示する。
